# Лабораторна робота №3: Налаштування CI/CD конвеєра для розгортання Docker-контейнеру

Цей проєкт містить автоматизований життєвий цикл розробки, збірки та розгортання вебзастосунку на базі Node.js за допомогою інструментів **Continuous Integration (CI)** та **Continuous Delivery (CD)** через GitHub Actions.

## 🚀 Архітектура системи
Проєкт розгорнуто на базі двох віртуальних машин у VirtualBox (ОС Ubuntu Server), які взаємодіють у локальній мережі через мережевий міст (Bridged Adapter):
1. **Runner Node (`192.168.0.143`)** — службовий вузол, на якому працює локальний GitHub Self-Hosted Runner. Він приймає команди від GitHub Actions та виконує деплой.
2. **Target Node (`192.168.0.142`)** — цільовий сервер, де розгортається застосунок у вигляді Docker-контейнера, налаштовано СУБД MariaDB та вебсервер Nginx як зворотний проксі (Reverse Proxy).

---

## 🛠 Компоненти автоматизації

### 1. Continuous Integration (CI)
Конвеєр автоматично запускається при кожному пуші у гілку `main` або створенні тегів і виконує такі етапи перевірки:
* **Аналіз Bash-скриптів:** Перевірка скриптів автоматизації за допомогою `Shellcheck`.
* **Лінтинг Dockerfile:** Контроль якості та безпеки Docker-образу за допомогою `Hadolint` (усі версії npm-пакетів жорстко зафіксовані).
* **Статичний аналіз коду:** Перевірка синтаксису та стилістики Node.js коду за допомогою `ESLint`.
* **Тестування:** Запуск автоматизованих Unit-тестів із генерацією звітів про покриття коду (Code Coverage) та завантаженням їх у вигляді артефактів GitHub.

### 2. Dockerization & Registry
Після успішного проходження тестів запускається етап збірки:
* Створюється Docker-образ на базі `node:18-alpine`.
* Образ автоматично публікується у сховище **GitHub Container Registry (GHCR)**.
* Налаштовано тегування: `latest` (для актуального стану `main`), `sha-*` (конкретний коміт) та `stable` (при створенні релізного Git-тегу).

### 3. Continuous Delivery (CD)
Повний процес розгортання запускається **виключно** при створенні анотованого тегу (наприклад, `v1.0.1`):
* GitHub Actions передає задачу на локальний **Self-Hosted Runner**.
* Ранер підключається до **Target Node** через SSH без запиту пароля (за допомогою SSH-ключів та безпарольного `sudo`).
* На цільовому сервері зупиняється старий сервіс, стягується новий образ (`stable`) з GHCR.
* Запускається автоматична міграція бази даних MariaDB за допомогою `migrate.js`.
* Застосунок розгортається у Docker з прапорцем `--network host` для прямої взаємодії з СУБД на хості.
* Керування життєвим циклом контейнера здійснюється через створений системний юніт **systemd** (`mywebapp.service`).

### 4. Верифікація (Verification)
Після запуску контейнера ранер автоматично здійснює перевірку:
* Тестує синтаксис конфігурації Nginx (`nginx -t`).
* Перевіряє доступність ендпоінту здоров'я застосунку через Nginx: очікується відповідь `HTTP 200` від `http://192.168.0.142/health/alive`.

## 📂 Структура проєкту
* `app/` — вихідний код Node.js застосунку, міграції, тести та конфіг `package.json`.
* `scripts/` — баш-скрипти для розгортання та файл конфігурації сервісу `mywebapp.service`.
* `.github/workflows/ci.yml` — повний опис конвеєра CI/CD (GitHub Actions).
* `Dockerfile` — інструкція для збірки Docker-образу.

## ⚙️ Як запустити новий реліз
Для внесення змін та автоматичного деплою нової версії застосунку:

1. Зробіть коміт та надішліть зміни до репозиторію:
   ```bash
   git add .
   git commit -m "Опис змін"
   git push origin main
   ```

2. Створіть анотований релізний тег версії:
   ```bash
   git tag -a v1.0.2 -m "Реліз версії 1.0.2"
   ```

3. Надішліть тег на GitHub для запуску CD:
   ```bash
   git push origin v1.0.2
   ```

Вкладка **Actions** автоматично відобразить процес розгортання та верифікації системи.
